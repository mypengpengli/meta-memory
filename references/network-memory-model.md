# 主题网络模型

人的记忆不是标准文件夹。
所以这套系统不能只做树状分类，必须是“树结构 + 主题网络”的混合模型。

## 一、为什么单纯树状分类不够

同一条记忆经常会同时属于多个方向。

例如：

- “Cloudflare 部署坑”既属于工作/编程，也属于某网站项目，也属于运维经验
- “new api thinking 模式问题”既属于 AI 工具，也属于接口中转，也属于配置踩坑
- “小说语言风格”既属于学习/写作，也属于个人偏好

如果只靠目录，你迟早会遇到放不下的问题。

## 二、解决方式：树结构 + 图关系

### 树结构负责什么

树结构负责：

- 人能看懂
- 文件好管理
- 大体上知道主归属领域

也就是说，一条记忆仍然要有一个主目录归属。

### 图关系负责什么

图关系负责：

- 交叉主题
- 关联项目
- 关联来源
- 标签命中
- 召回时的横向联想

## 三、每条记忆建议都带这些字段

哪怕先不用数据库，文档头部也应该尽量保留这些字段：

- `area`
  主领域，例如 `work`
- `memory_kind`
  固定 / 领域 / 主题 / 项目 / 会话 / 候选 / 归档
- `topic`
  主主题
- `subtopics`
  子主题列表
- `tags`
  标签
- `related_topics`
  关联主题
- `related_projects`
  关联项目
- `related_sources`
  关联来源
- `stability`
  稳定程度
- `load_hint`
  常驻 / 按主题 / 按项目 / 按会话 / 仅检索

## 四、这样做的好处

表面上你仍然可以按目录组织：

- 工作
- 学习
- 家庭与子女教育
- 日常生活

但索引层可以做交叉关联：

- 一个主题跨多个领域
- 一个项目关联多个主题
- 一个来源可以支撑多条记忆

## 五、第一版怎么落地最稳

第一版不要急着做复杂图数据库。

先这样就够：

- 目录负责主归属
- Markdown frontmatter 负责关联字段
- 以后如果量变大，再加 SQLite 索引层

这样既可移植，也方便以后扩展。
